# 任务五：AI 使用记录与人工判断

## 一、我们的 AI 使用方式：交叉审核，而非单点问答

我们没有把代码丢给一个 AI 问"有什么问题"，而是**并行派出 4 个独立审查者，各自只看一个维度，最后由人工收口**。理由是：单个 AI 的一次评审，你无法区分"它没找到"和"确实没有"。多个独立视角的**交集是高置信区，差集是需要人工裁决的区域**——差集恰恰是最有价值的部分。

| 审查者 | 模型 | 分工 | 产出 |
| --- | --- | --- | --- |
| A | Claude | 可理解性 / 命名 / 职责 / 复杂度 / 一致性 | 40 条 |
| B | Claude | 健壮性 / 输入校验 / 安全 / HTTP 协议 / 资源管理 | 29 条 |
| C | Claude | 可测试性 / 依赖隔离 / 全局状态 / 日志 / 并发 | 25 条 |
| D | **Codex (GPT-5)** | 全维度独立交叉审查 | 25 条 |
| 人工 | — | 复现、裁决、去重、定稿 | **30 条**（合并后） |

**跨模型是刻意的**：A/B/C 同源，有共同盲区；D 换一个模型家族，用来检验前三者的共识是不是"同一种偏见的三次复读"。事实证明有效——见下文 §4.1。

## 二、使用的提示词

### 提示词 1：分维度审查（对 A/B/C，此处摘 B）

```text
你是一名资深 Web 安全与后端健壮性专家，正在评审一份手写的 asyncio HTTP 服务器。

请只聚焦【健壮性 / 输入校验 / 错误处理 / 安全 / HTTP 协议正确性 / 资源管理】维度，
做穷尽式审查。不要写重构代码，只出问题清单。

对每个问题给出：
1. 精确位置（file:line）
2. 问题类型
3. 可触发的具体场景：什么输入 → 什么后果（要能实际复现，不要臆测）
4. 严重程度（高/中/低）及理由
5. 修改建议

必须核实的疑点（自己动手验证，不要照抄我的判断）：
- /api/v1/sleep 中 source.get("sec") 返回 None 时 len(None) 会怎样？
- safe_static_path 的路径穿越防护是否真的有效？符号链接呢？
- do_client 里 except Exception 捕获一切并回 400，是否把 500 类错误伪装成客户端错误？
- 会话固定：登录成功后是否重新生成 sid？
- SESSIONS 全局字典无上限：是否内存无限增长？
（…共 12 条疑点…）

请务必自己找更多，不要局限于以上清单。
```

三个设计要点：

1. **限定单一维度**。让一个 AI 同时看 9 个维度，结果是每个维度都浅尝辄止。分维度后，A 在命名维度挖到了 40 条——远超"综合评审"能给的深度。
2. **"要能实际复现，不要臆测"**。这一句是整份提示词里最重要的。没有它，AI 会大量输出"可能存在""建议关注"这类无法证伪的话。
3. **给出疑点，但明确要求"自己动手验证，不要照抄我的判断"**。我们故意在疑点里混入了几条**我们自己也不确定、甚至怀疑是错的**（如路径穿越）。如果 AI 全盘复述我们的怀疑，说明它在迎合；如果它敢反驳，说明它真的验证了。**这是一个针对 AI 谄媚倾向的陷阱，而它确实起作用了。**

### 提示词 2：跨模型交叉审查（对 D / Codex）

```text
请用 Codex 对以下文件做一次独立、彻底的代码评审。
审查维度（全覆盖，不要只挑一两个）：命名 / 职责 / 可测试性 / 错误处理 /
输入校验 / 日志 / 安全 / HTTP 协议正确性 / 维护风险。

要求：
- 每个问题给出 file:line、问题类型、可复现的触发场景、严重度、修改建议
- 重点找真实 bug（能触发异常或错误行为的），不要只列风格偏好
- 不要修改任何代码，只输出评审报告
```

**刻意不告诉 D 前三个 Agent 找到了什么**——否则它会变成"复核"而不是"独立发现"，交叉验证的意义就没了。

## 三、AI 给出了哪些建议（汇总）

四份报告合计 119 条，去重后 30 条。分布：

- **真实缺陷 / 可复现**：15 条（sleep 崩溃、会话固定、过期会话复活、slowloris、截断请求、CL 校验、早退丢 cookie…）
- **结构与可维护性**：11 条（do_cookie 双职责、route 巨型分支、解析器副作用、两份头解析…）
- **风格排版**：4 条

四份报告**独立收敛到同一个根因**："这个文件没有层边界，所以职责必然漂移、命名必然退化、一致性必然瓦解、测试必然失效。"这个共识度让我们有信心把它写进任务一的结论。

## 四、采纳了哪些（及为什么）

### 4.1 D（Codex）独家发现：过期会话被复活 —— **采纳**

四个审查者里**只有 Codex 发现**：`get_session` 命中后先刷新 `last_seen`（L116），而 `cleanup_sessions()` 在 `route()` 里更晚才跑（L303）。**判定过期发生在续期之后，所以过期会话永远不会被清掉。**

我们复现确认后采纳，写成 `test_expired_session_is_not_revived_on_access`。

> **这条单独证明了跨模型审查的价值。** 三个 Claude 都盯着 `SESSIONS` 无上限（问题 7），却集体漏掉了紧挨着的这个顺序 bug。它不是"没找到的第 31 条"——它是三个同源模型的**共同盲区**。如果我们只用一个模型、或只用同源的多个实例，这条会永远漏掉。

### 4.2 A 独家发现：早退分支绕过 Set-Cookie 兜底 —— **采纳**

A 实测 `DELETE /api/v1/sleep` → 405 且**无 Set-Cookie**，而 `GET /nope` → 404 **有** Set-Cookie。这不是风格问题，是行为分裂。采纳，并且**改变了我们的重构方案**：Cookie 兜底从"route 函数尾部"上提到"管线出口"，让任何分支都无法绕过。

### 4.3 C 独家发现：测试套件的证据能力已失效 —— **采纳，且改变了整个测试策略**

C 连续跑三次原测试，得到三种结果，其中一次 7/7 全绿而被测的是残留的旧孤儿进程。

这条我们评为**整份练习最有价值的发现**，因为它把"可测试性"从一个抽象的美学要求，变成了一个可量化的事实：**测试信号与代码正确性脱钩了**。它直接决定了我们的测试用 `port=0` 而非 8080，且 `start()` 必须同步绑定完再返回。

### 4.4 A 独家发现：HTML 缩进泄漏进响应体 —— **采纳**

`.strip()` 只去首尾不做逐行 dedent，`route()` 里 12 空格缩进的 HTML 每行都多 12 个空格发出去了。真正的风险 A 也说清了：**日后任何人把这段 HTML 搬进函数（缩进变化）都会静默改变响应字节**，而失败原因与改动意图毫无关联。采纳，模板提为模块常量。

### 4.5 B/D 共同发现的 framing 问题 —— **采纳**

截断请求被当完整请求、`Content-Length: -1` 绕过上限、chunked 被忽略。B 和 D 独立发现且复现路径一致，高置信，全部采纳并写了 raw socket 回归测试。

## 五、拒绝了哪些（及为什么）——本节是重点

**AI 报错的比例不低，而且错得很像真的。** 以下 6 条我们逐一动手复现后驳回：

| # | AI 的说法 | 谁说的 | 人工裁决 | 依据 |
| --- | --- | --- | --- | --- |
| 1 | `safe_static_path` 有路径穿越漏洞 | 初版提示词里我们自己的怀疑 | **驳回** | 实测 `../`、`%2e%2e`、`//etc/passwd` 全部 403。`STATIC_ROOT` 已 resolve，`Path / "/abs"` 会被 `resolve()`+`relative_to()` 拦下 |
| 2 | 符号链接可绕过静态目录限制 | 同上 | **驳回** | 实建 `static/escape_link -> /etc/passwd`，仍 403。`resolve()` 跟随符号链接，逃逸后 `relative_to` 必然失败 |
| 3 | 解析失败会泄漏一个 session | B 报告倾向认为会 | **驳回** | C 实测：`get_session` 在 L171，位于所有 `raise`（L139/L148）**之后**，解析失败后 `SESSIONS` size = 0 |
| 4 | sid 可做 CRLF 响应头注入 | B | **降级为潜伏风险** | 实测注入后响应头无 `X-Injected`——因为 `get_session` 仅当 sid 已是 `SESSIONS` 的 key 才回显，而 key 只由 `token_urlsafe` 生成。当前不可利用 |
| 5 | `int('9'*100000)` 大整数 DoS | B | **驳回** | CPython 4300 位限制使其直接抛 ValueError，走正常 400 分支 |
| 6 | `visits` 读-改-写存在并发竞态 | C 初判 | **改判为"今天正确，但纯属侥幸"** | 实测 100 并发 → visits=100（正确），因为中间恰好没有 `await`。但 C 又实测：**加一个 await 立刻变成 visits=1** |

### 值得展开的两条

**第 1、2 条（路径穿越）**：这是最典型的"AI 顺着你的怀疑往下编"。我们在提示词里问"路径穿越防护是否真的有效？符号链接呢？"——如果 AI 迎合，它会顺势报一个漏洞。**三个独立审查者都顶住了，都明确回答"防护有效"**，其中 B 还主动建了符号链接做实验。三方一致 + 我们自己的验证，四方独立得出同一否定结论，我们才敢把它写进"经核实不成立"。

> **教训**：AI 报出的漏洞要复现，AI **没报**的漏洞更要警惕，但 AI **顶住压力说"这里没问题"**的结论，可信度反而是最高的——因为它逆着提问者的预设。

**第 6 条（并发竞态）**：这条最微妙。C 一开始按"读-改-写必有竞态"的经验报了，但我们要求复现时，它实测发现**今天是对的**。如果止步于此，这条就该删掉。但 C 多做了一步：**给 `get_session` 加一个 `await` 再测，visits 立刻从 100 掉到 1。**

我们最终没把它列为"当前缺陷"，而是列为"潜伏缺陷"，并在任务一里写明：**它今天是对的，但对得没有道理**——正确性来自"恰好没有 await"这个偶然事实，代码里没有任何注释或结构在保护这个不变量。任何一次自然演进都会让 99% 的计数凭空蒸发，且不报错、无日志、测试测不出。

这是我们认为最能体现"人工把关"价值的一条：**AI 的初判（有竞态）错了，AI 的复现（没竞态）也不完整，真正的答案在两者之间。**

## 六、人工最终把关点

### 6.1 一票否决：不能复现的结论一律不进清单

我们对 119 条的第一道筛子就是"**你能让它在跑着的服务器上发生吗**"。这一条砍掉了大量"建议关注""可能存在"式的输出。

进清单的每一条都附了复现路径。任务一里"经核实不成立"那一节，就是这道筛子的产物——**我们认为它和问题清单本身同样重要**，因为它记录了 AI 会在哪里出错。

### 6.2 人工把三个人的"小插曲"拼成了一条根本性缺陷

验证过程中出现过一个矛盾现象：服务器日志里有 `Address already in use` 的 traceback，但 curl 请求又全部成功。查下去才发现，8080 已被另一个进程占用，我们启动的实例 bind 失败退出，那些响应来自别处。

有意思的是，**审查者 B 独立踩了同一个坑**，并在报告里主动声明它的前几次测试"silently hit the wrong server and produced bogus results"，随后改在空闲端口上重跑。

而审查者 C 没有把它当成自己的实验事故，而是**升级成了对原测试套件的根本性指控**（§4.3）——同一个现象，从"我的环境有问题"变成了"这份代码的测试策略是坏的"。

**人工在这里的价值是把三个碎片拼起来**：我的困惑 + B 的自我修正 + C 的系统性指控，合起来才是任务一的问题 1。单独看，每一条都只是个小插曲；拼起来才看清，这不是环境的错，是代码把测试逼到了必须依赖全局端口的绝境。

### 6.3 人工驳回了 AI 的一条方案级建议

B 建议 SESSIONS 改为"**惰性创建**：只在真正需要存状态时才落库"。技术上这是更彻底的修复。

**我们没有采纳**，理由是：它会改变"每个响应都带 sid"这个既有行为，而这属于**业务行为变更，不该混在风格重构里**。我们退一步只加容量上限 + LRU 淘汰（同样堵住了内存无限增长），把惰性创建显式登记在 `02-重构方案.md` §9"我们没有做的事"里。

> **AI 倾向于给"最优解"，但重构的约束是"不改变业务行为"。最优解和当前该做的事，不是一回事。**

### 6.4 人工抓到了自己写的重构代码里的 bug（AI 没抓到）

我们在写 `ConnectionHandler` 时，一开始把 `_method` / `_path` / `_include_body` 存成了**实例属性**用于日志。但 `ConnectionHandler` 实例被所有并发连接共享——**并发请求会互相覆盖这些字段**。

这是一个真 bug，而且讽刺的是：它正是我们在任务一里批评原代码的那类问题（共享可变状态）。**四个 AI 都没有审到这段代码**（它们只审原代码，不审我们的重构产物）。

我们自己发现后改成了 `Exchange` 值对象随调用栈传递，并加了注释说明为什么不能存实例上。

> **教训：AI 审的是你给它的代码。你新写的代码，没人审。**

### 6.5 人工修正了自己写错的测试

`test_cleanup_expired_is_throttled_by_interval` 第一次跑挂了。我们先看的是"代码是不是错了"，查下来发现——**是测试错了**：我们在断言"被节流跳过"之前又把时钟推进了 3601 秒，早超过 60 秒的节流窗口。

我们认为这个插曲值得记录：**测试挂了，第一反应不该是改代码去迁就测试，也不该是改测试去迁就代码，而是搞清楚谁错了。**

### 6.6 我们如何判断重构后的代码质量

不靠"感觉更清晰了"，靠四条可检验的标准：

| 标准 | 检验方式 | 结果 |
| --- | --- | --- |
| 行为没被改坏 | 原始测试文件**逐字节未改**（md5 一致）跑重构版 | 7/7 通过 |
| 确实测的是新代码 | 用只有重构版才有的行为交叉确认（无参 sleep 返回 200、首页无 password） | 确认 |
| 解耦是真的 | **单元测试能不能写出来**——38 个不起进程、不绑端口的用例 | 0.06s 通过 |
| 缺陷不会回来 | 16 个回归用例逐条锁定已修复的缺陷 | 通过 |

第三条是我们最看重的。**"可测试性"不是一个形容词，它是一个二值事实：这个测试你写不写得出来。** 原代码上 `test_parse_request_has_no_side_effects_on_sessions` 这个断言根本无法通过——不是难写，是不可能。它现在通过了，这就是解耦完成的证据。

## 七、下次用 AI 写代码，我们会额外检查什么

1. **要求复现，不接受"可能存在"**。一句"要能实际复现，不要臆测"让报告质量发生质变。
2. **换个模型家族再审一遍**。同源模型有共同盲区——过期会话复活那条，三个 Claude 集体漏掉，Codex 一眼看到。
3. **在提示词里埋一条自己也不确定的疑点**，看 AI 敢不敢反驳。敢反驳的报告，可信度高一个量级。
4. **AI 审不到你新写的代码**。我们自己的重构里就藏了一个共享可变状态的 bug。
5. **AI 的"最优解"未必是当前该做的事**。它不知道"不改变业务行为"这条约束有多硬。
6. **测试全绿不等于代码是对的**——先问"这套测试到底在测什么"。这是这次练习给我们最大的教训。
