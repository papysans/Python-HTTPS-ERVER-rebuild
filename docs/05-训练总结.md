# 训练总结

## 一、这次重构中，最重要的一个改动是什么？

**把会话的创建从 `parse_request()` 里搬出来。**

这个改动本身只有几行，但它是整次重构的枢纽。原代码是这样：

```python
def parse_request(raw: bytes) -> Request:     # 签名承诺：字节 → 结构体，纯函数
    ...
    sid, session, is_new = get_session(request)      # ← 向全局 SESSIONS 写入
    request.session["visits"] = request.session.get("visits", 0) + 1   # ← 副作用计数
```

一个叫"解析"、签名是 `(bytes) -> Request` 的函数，实际上改写了模块级全局状态。

我们最初以为这只是"职责不单一"，一个风格问题。**但顺着它往下拽，拽出了一整条因果链**：

```
解析器带副作用
  → 解析与会话生命周期焊死
  → 想测「解析」就必须先懂会话子系统、先 SESSIONS.clear()
  → 两者都无法单独测
  → 核心逻辑只能从网络入口触达
  → 测试被迫写成进程级黑盒，硬编码打 8080
  → 端口被别的进程占用时，测试连上占位者
  → 7/7 全绿，而被测的是一个早该死掉的旧进程
  → 测试信号与代码正确性彻底脱钩
```

**从"函数名撒了个谎"到"整套测试失去证据能力"，中间只隔了六步。**

而且它的危害不止于测试。因为计数写在了解析层，**业务层根本没有机会决定什么才算一次访问**——于是 404、favicon、静态文件全都计入了 `visits`，首页展示给用户的"访问次数"语义是错的。同时每个匿名请求都会永久占用一条会话，1 小时不回收，构成一条无需认证、成本极低的内存耗尽路径。

**一个"顺手多做一点"的解析函数，同时产出了：一个语义错误、一个 DoS 面、和一套失效的测试。**

我们选它作为最重要的改动，不是因为改动量大，而是因为它教会我们一件事：**职责不单一从来不是审美问题。它是一个会沿着依赖链向外传播的结构性缺陷，最终一定会在某个你意想不到的地方结账。**

## 二、哪个问题最能体现"代码能跑不等于代码质量好"？

**测试 7/7 全绿，而被测的是一个残留的旧孤儿进程。**

这份代码"能跑"的证据非常充分：功能完整，7 个功能测试全部通过，跑起来 curl 什么都对。按传统标准，它已经交付了。

但我们连续跑三次同一套测试，得到三种结果：

| 运行 | 结果 | 实际被测对象 |
| --- | --- | --- |
| 第 1 次 | 4 失败 / 3 通过 | 前 4 个打到外部程序，后 3 个打到子进程 |
| 第 2 次 | **7 全部通过（0.06s）** | **一个残留的旧孤儿进程** |
| 第 3 次 | 7 全部失败（0.05s） | 全部打到本机另一个占用 8080 的程序 |

第 2 次是最恐怖的：**你可以把 `route()` 整个函数删掉，测试照样 7/7 全绿。**

它之所以最能说明问题，是因为它把"代码能跑不等于代码质量好"又往前推了一层：

> **不只是"能跑 ≠ 质量好"，而是"测试全绿 ≠ 代码是对的"。**

我们平时对抗"能跑就行"的武器就是测试。而这个案例说明：**当可测试性缺失时，测试本身也会沦为"看起来能跑"的一部分。** 你以为你有安全网，其实你只有一张安全网的照片。

它的因果链是唯一的，没有任何偶然成分：

- 核心逻辑耦合全局状态 → 只能从 8080 这一个入口触达
- → 被迫写进程级 e2e → 依赖一个全局的、不可控的资源（端口）
- → `wait_for_server` 先 `poll()` 后 `connect`，子进程启动的几十毫秒窗口里，`poll()` 必然返回 `None`，而 `connect` 连上了占位者
- → 子进程带着 `EADDRINUSE` 崩溃，栈被灌进 `subprocess.PIPE` 后**永远不会被打印**
- → 测试对着一个不相干的进程跑完全程，报告全绿

**每一环都是前一环的必然结果。可测试性缺失不是"不方便"，它直接摧毁了测试的证据能力。**

排第二的是那行死代码：

```python
sec = source.get("sec")   # 缺参时返回 None
sleep_sec = 1             # 这行永远不会执行
if len(sec) > 0:          # len(None) → TypeError
```

`sleep_sec = 1` 想表达"默认睡 1 秒"，但**它想兜底的那个场景，恰恰就是崩溃的场景**。这行代码是一句谎言，而且它被 `except Exception` → 400 完美地藏了起来：一个服务端 `TypeError`（500 级缺陷）被报告成 `400 Bad Request`。**任何监控 5xx 率的告警永远不会响——服务在坏，指标却是绿的。**

## 三、下次使用 AI 写代码，我会额外检查哪些质量风险？

### 1. 先问"这个函数的名字，和它实际干的事，是同一件事吗"

AI 生成的代码最典型的坏味道不是写得丑，是**名实不符**：`parse_request` 会建 session、`get_session` 会续期、`read_http_request` 会解析头、`do_cookie` 靠一个布尔参数身兼解析与序列化两职（还把返回类型注解写成 `-> str`，而解析分支实际返回 `dict`）。

**签名一旦撒谎，类型检查器、IDE、调用图分析、你的直觉，全部失效。** 而这恰恰是 AI 最容易犯的错——它按"看起来该有的样子"命名，按"能跑就行"实现。

### 2. 检查有没有"层"

这次 30 个问题只有 1 个根因：**没有层边界**。

AI 很擅长写出一个个单独看都合理的函数，然后把它们平铺在一个命名空间里。没有墙，捷径就一定会被走：`parse_request` 能摸到 `SESSIONS`，它就摸了；`route` 能调 `cleanup_sessions`，它就调了。

**下次的检查动作很具体**：能不能一句话说清每个模块的边界？有没有哪个模块 import 了它不该认识的东西（比如业务层 import socket）？

### 3. 把"我能为它写单元测试吗"当作硬指标

**"可测试性"不是形容词，是二值事实。**

判断标准：不起进程、不绑端口、不碰全局状态，能不能写出一个断言？写不出来，就说明依赖是"伸手去拿"（reach out）而不是"传进来"（inject）的。

这次我们用 `test_parse_request_has_no_side_effects_on_sessions` 作为解耦的验收标准——这个断言在原代码上**不是难写，是不可能通过**。它现在过了，解耦就是真的。

### 4. AI 报的漏洞要复现；AI 顶住压力说"没问题"的，反而最可信

我们在提示词里故意埋了几条自己也不确定的疑点（"路径穿越防护是否真的有效？符号链接呢？"）。如果 AI 迎合提问者的预设，它会顺势编一个漏洞出来。

**三个独立审查者都顶住了**，都明确回答"防护有效"，其中一个还主动建了符号链接做实验。四方独立得出同一否定结论，这条我们才敢下定论。

反过来，AI 主动报的 6 条我们全部复现后驳回了——包括大整数 DoS（CPython 4300 位限制直接挡住）、解析失败泄漏 session（`get_session` 在所有 `raise` 之后）。**错得都很像真的。**

### 5. 换一个模型家族再审一遍

**同源模型有共同盲区，而且是集体性的。**

"过期会话被复活"（`get_session` 先刷新 `last_seen`，`cleanup_sessions` 后判定过期，导致过期策略形同虚设）——三个 Claude 都盯着紧挨着的 `SESSIONS` 无上限问题，**集体漏掉**了这个顺序 bug；Codex 一眼看到。

它不是"没找到的第 31 条"，它是**三个同源实例的共同盲区**。多派几个同样的 AI，只会让你对同一种偏见更有信心。

### 6. 记住：AI 审的是你给它的代码，你新写的代码没人审

我们的重构里藏了一个真 bug：`ConnectionHandler` 把日志字段存成实例属性，而该实例被所有并发连接共享——**并发请求会互相覆盖**。

讽刺的是，这正是我们在任务一里批评原代码的那类问题（共享可变状态）。四个 AI 都没审到它，因为它们只审原代码。**是我们自己在写的时候发现的。**

### 7. 测试挂了，先搞清楚是谁错了

我们有一个测试第一次跑挂了。第一反应不该是改代码迁就测试，也不该是改测试迁就代码。查下来是**测试写错了**——我们在断言"被节流跳过"之前又把假时钟推进了 3601 秒，早超过 60 秒的节流窗口。

### 8. 风格重构与行为变更，必须分开

这次我们识别出 15 处"修复必然改变行为"的地方（sleep 崩溃、会话固定、HEAD 返回 body……）。它们**每一条都是缺陷**，但"修缺陷"和"改风格"是两件事。

我们的做法是列一张显式的《行为变更清单》，逐条登记原行为、新行为、理由。**而不是混在重构里静默改掉**——否则日后出问题，没人知道是哪次重构动了业务语义。

同样重要的是那张《我们没有做的事》清单（CSRF、登录限速、静态文件流式传输、会话惰性创建）。**诚实登记范围之外的项，比假装全都解决了更有价值。**

---

## 四、一句话

这份代码最深的问题，不在它写错的任何一行里，而在它**没有的那些墙**里。

墙没了之后，一切都是必然的：职责必然漂移，命名必然退化，一致性必然瓦解，测试必然失效。终点就是我们实测到的那一幕——

**7/7 全绿，0.06 秒，测的是一个早就该死掉的进程。**
