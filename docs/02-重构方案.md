# 任务二：重构方案

## 一、总体思路：先划层，再改名

任务一的结论是：多数可维护性与可测试性问题共享一个主要结构性根因——**缺少清晰的层边界**。因此我们不按问题清单从上往下逐条改，而是先把墙立起来，再分别处理安全、协议和资源治理问题。

原代码 426 行全部挤在一个文件里（这是原始编码任务的要求）。我们拆成 5 个模块，**层间只有一个契约：`Request` / `Response`**。

```
http_server.py   ← 唯一接触 socket 的地方：连接生命周期、读写超时、并发上限
      ↓ Request
handlers.py      ← 路由表 + 各端点业务；纯逻辑，不认识 socket
      ↓ 依赖注入
sessions.py      ← SessionStore：会话的查/建/续期/淘汰
http_core.py     ← 报文解析与序列化；全部纯函数
config.py        ← 配置值对象；不再有模块级全局常量
```

判断标准很简单：**`http_server.py` 之外的任何模块，都不应该 import socket。**

## 二、拆分了哪些函数，各自职责

| 原函数 | 拆分去向 | 新职责（一句话） |
| --- | --- | --- |
| `do_cookie(do_parse, ...)` | `parse_cookie_header()` | 把 Cookie 请求头解析成 dict |
| | `build_set_cookie()` | 构造 Set-Cookie 值，并拒绝 CRLF 注入 |
| `response_type1()` | `text_response()` | 构造 text/plain 响应 |
| `response_type2()` | `html_response()` | 构造 text/html 响应 |
| `read_http_request()` | `read_request()` | 只负责从 socket 读字节，读完即交给解析 |
| | `parse_head()` | 解析请求行 + 头（**全项目唯一一份**） |
| | `parse_content_length()` | 校验 Content-Length 与 Transfer-Encoding |
| `parse_request()` | `build_request()` | 由已解析的头 + body 组装 Request（**纯函数，不建会话**） |
| `get_session()` ×3 职责 | `SessionStore.get()` | 只查，且过期即视为不存在 |
| | `SessionStore.create()` | 建会话，超容量时 LRU 淘汰 |
| | `SessionStore.touch()` | 续期 |
| | `SessionStore.rotate_id()` | 换 sid 保数据（防会话固定） |
| `cleanup_sessions()` | `SessionStore.cleanup_expired()` | 按间隔节流的过期清理 |
| `route()` 的分发部分 | `Router.dispatch()` | 查路由表并委托 |
| `route()` 的 sum 内联逻辑 | `parse_int_list()` | 字符串列表 → 整数列表 / 首个非法项 |
| `route()` 的 sleep 内联逻辑 | `parse_sleep_seconds()` | 解析并校验 sleep 秒数 |
| `route()` 的三元式 ×2 | `extract_params()` | POST 有表单取表单，否则取查询串 |
| `route()` 的首页内联 HTML | `INDEX_TEMPLATE` 常量 | 模板与代码缩进解耦 |
| `route()` 尾部 Cookie 兜底 | `_attach_session_cookie()` | 移到管线出口，**分支无法绕过** |
| `safe_static_path()` | `resolve_static_path(root, path)` | root 变参数，可脱离真实目录测 |
| `do_client()` | `ConnectionHandler.handle()` | 一条连接的完整生命周期 |
| `serve()` | `HttpServer.start()` / `.serve_forever()` | 绑定与运行分离，端口可注入 |

按当前代码统计，所有函数均不超过 50 行，参数不超过 5 个，嵌套不超过 3 层。

## 三、隔离了哪些外部依赖

原代码的依赖全是在函数内部"伸手去拿"（reach out），我们一律改为从外部传入：

| 依赖 | 原来 | 现在 |
| --- | --- | --- |
| 会话存储 | 模块级 `SESSIONS` 字典 | `SessionStore` 实例，构造注入 |
| 时间 | 函数内直接 `time.time()` | `clock` 可调用对象注入 → 测试传假时钟，不必真等一小时 |
| sid 生成 | 函数内直接 `secrets.token_urlsafe` | `id_factory` 注入 → sid 可预测，断言可确定 |
| 静态根目录 | import 期 resolve 的 `STATIC_ROOT` | `ServerConfig.static_root` → 测试指向 tmp_path |
| 凭据 | `USERNAME`/`PASSWORD` 模块常量 | `ServerConfig` 字段 |
| sleep | 直接 `asyncio.sleep` | `Router(sleep=...)` 注入 → 测试不必真的睡 |
| 监听端口 | 模块常量 8080 | `ServerConfig.port`，支持 `port=0` |
| socket | 业务函数直接吃 socket | 只有 `http_server.py` 认识 socket |

**这一列的每一行，都直接对应一类原本写不出来的测试。**

## 四、校验了哪些输入

原则：**协议级输入在进入路由层前完成校验；端点业务参数由对应 handler 在产生副作用前完成校验。**

| 输入 | 校验规则 | 不合规时 |
| --- | --- | --- |
| 请求行 | 必须是三段式 | 400 |
| HTTP 版本 | 仅 `HTTP/1.1` / `HTTP/1.0` | 505 |
| 请求头 | 必须含冒号 | 400 |
| 头部总大小 | ≤ 16 KB | 431 |
| `Content-Length` | 必须为非负十进制整数 | 400 |
| `Content-Length` 重复且冲突 | 拒绝（RFC 9112 §6.3） | 400 |
| `Transfer-Encoding` | 不支持则明确拒绝 | 501 |
| body 实际长度 | 必须等于声明长度 | 400 |
| body 大小 | ≤ 1 MB | 413 |
| `sec` 参数 | 整数、非负、≤ 10 秒 | 400 + 结构化 JSON |
| `numbers` 参数 | 每项均为整数 | 400 + 结构化 JSON |
| 静态路径 | resolve 后必须在 static_root 内 | 403 |
| 响应 Cookie 名/值 | 序列化时不得含 CR/LF | 抛 ValueError |

## 五、统一了哪些错误处理

原代码有**四套**错误表达并存（JSON / text/plain / HTML / 异常原文），且所有异常一律映射为 400。我们统一错误分类和异常映射规则；具体响应格式仍按端点类型区分：

```
HttpError(status, reason, client_message, log_message)
├── BadRequest        400
├── PayloadTooLarge   413
└── RequestTimeout    408
```

关键设计：**`client_message` 与 `log_message` 分离**。前者回给客户端，只包含受控信息，不含异常类型、堆栈或服务器路径；后者只进日志，可以包含定位所需的细节。这一条直接消除了问题 3 的信息泄露。

捕获策略分三层：

- `HttpError` → 按其自带状态码响应，记 WARNING
- 其它 `Exception` → **500**（不再是 400），记 `logger.exception()` 含完整堆栈，响应体固定为 `internal server error`
- 超时 → 408

> 这条最重要：**服务端 bug 必须表现为 5xx。** 原代码把 500 报成 400，监控 5xx 率的告警永远不会响。

## 六、补充了哪些关键日志

全面改用 `logging`（替代 3 处 `print`），并在这些位置补日志：

| 位置 | 级别 | 内容 |
| --- | --- | --- |
| 每个请求结束 | INFO | 客户端 IP、方法、路径、状态码、耗时 |
| 外部输入非法 | INFO | 具体哪个字段、什么值 |
| 认证失败 | **WARNING** | 用户名（**使暴力破解可检测**） |
| 路径穿越尝试 | **WARNING** | 原始 URL（**使安全探测可见**） |
| 读超时 / 报文被拒 | WARNING | 客户端地址 + 原因 |
| handler 崩溃 | **ERROR** | 完整堆栈 |
| 静态文件读失败 | ERROR | 路径 + errno |
| 会话过期清理 | INFO | 清理条数 + 剩余 |
| 会话表满触发淘汰 | WARNING | 上限值 |

## 七、准备补充哪些单元测试

见 `03-测试用例说明.md`。两层：

- **单元测试**（38 个）：不起进程、不绑端口、注入假时钟与假 sleep。原结构难以编写稳定、隔离且无需清理全局状态的单元测试。

## 八、刻意的行为变更清单

练习要求"不改变原始业务行为"，但任务一发现的部分"行为"**本身就是缺陷**。修 bug 必然改变行为。我们的原则是：**风格重构与行为变更必须分离，且每一处变更都要显式登记，而不是混在重构里静默改掉。**

| # | 端点/场景 | 原行为 | 新行为 | 理由 |
| --- | --- | --- | --- | --- |
| 1 | `GET /api/v1/sleep` 无参数 | 400 崩溃 | 200，默认 1 秒 | 原行为是 `len(None)` 崩溃，代码本意就是默认 1 秒 |
| 2 | `sec=abc` | 400 + 异常原文 | 400 + `{"error": "not an integer: abc"}` | 消除信息泄露，与 sum 的错误格式对齐 |
| 3 | `sec=-5` | 200 `{"sec": -5}` | 400 | 静默接受负数是语义错误 |
| 4 | `sec=99999999` | 200，钉住连接数年 | 400 | DoS |
| 5 | 服务端内部异常 | 400 + 异常原文 | 500 + 固定文案 | 服务端 bug 必须表现为 5xx |
| 6 | 登录成功 | 沿用原 sid | 轮换 sid | 会话固定可提权 |
| 7 | 首页 | 渲染 `admin/password` | 不渲染 | 向匿名者泄露凭据 |
| 8 | 早退分支（405 等） | 无 Set-Cookie | 有 Set-Cookie | 原行为使会话静默丢失 |
| 9 | `HEAD` 请求 | 返回实体 body | 仅头部，保留 Content-Length | RFC 9110 §9.3.2 |
| 10 | 截断请求 / body 不足 | 当作完整请求处理 | 400 | 凭空补造客户端未发送的数据 |
| 11 | `Content-Length: -1` / `abc` | 被接受 / 崩溃 | 400 | framing 攻击面 |
| 12 | `Transfer-Encoding: chunked` | 静默忽略 | 501 | 请求走私面 |
| 13 | `HTTP/9.9` | 200 | 505 | 协议合规 |
| 14 | 过期会话 | 被访问即复活 | 判定过期在刷新之前 | 过期策略原本形同虚设 |
| 15 | 首页 HTML | 每行多 12 空格 | dedent | 代码缩进泄漏进响应体 |

**未变更（刻意保留）**：`sum` 的四个别名 `numbers/nums/values/n`。它们无文档、无需求依据，但可能已有调用方依赖。我们保留行为，但把它提为具名常量 `SUM_PARAM_ALIASES` 并加注释说明这是历史包袱——**先让隐藏契约变得可见，再谈删除**。

## 九、我们没有做的事

诚实登记范围之外的项，避免让人误以为已经解决：

- **CSRF 防护**：`/logout` 仍接受 GET。需要引入 per-session token，超出本次重构范围。
- **登录限速**：无失败计数与退避。
- **密码哈希**：仍是明文常量比较（已改用 `compare_digest` 消除时序差），因为原始编码任务明确要求硬编码凭据。
- **静态文件流式传输**：仍是一次性 `read_bytes()`。已加 `X-Content-Type-Options`，但大文件的 2 倍内存放大未解决。
- **keep-alive**：仍是 `Connection: close`，与原代码一致。
- **会话惰性创建**：仍是每请求必建会话（仅加了容量上限 + LRU）。惰性创建是更彻底的修复，但会改变"每个响应都带 sid"的既有行为，我们选择先加上限，把它列为后续项。
