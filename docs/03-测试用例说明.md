## 一、先说最重要的一件事：为什么原代码只能有黑盒测试

原代码有 7 个功能测试，**全部通过**。但它们全是进程级黑盒 e2e：`subprocess` 拉起真进程、硬编码打 `127.0.0.1:8080`，**从不 `import http_server`**。

这不是作者偷懒，是**结构强迫的结果**：核心逻辑全部耦合模块级全局（`SESSIONS` / `time.time()` / `STATIC_ROOT` / `USERNAME`），唯一能触达它们的入口就是网络端口。于是测试只能从 8080 打进去。

代价我们在任务一里实测到了：**同一份代码连续跑三次，三种结果**，其中一次 7/7 全绿而被测的是一个残留的旧孤儿进程。

所以我们的测试策略是两层，而**第一层的存在本身就是解耦的验收标准**：

| 层 | 数量 | 起进程 | 绑端口 | 耗时 | 能否并行 |
| --- | --- | --- | --- | --- | --- |
| 单元测试 | 103 | 否 | 否 | 0.27s | 能 |
| 功能测试 | 25 | 否（进程内） | port=0 内核分配 | 1.4s | 能 |
| ~~原黑盒测试~~ | 7 | 是 | 硬编码 8080 | 0.06s~5s | **不能** |

功能测试用 `port=0` 而非 8080，并且 `HttpServer.start()` **同步完成绑定后才返回地址**——因此不存在"先 poll 再 connect"的竞态，**不可能出现连上了别人的服务却报告自己全绿**。

## 二、行为保持验证

`test_functional.py` 的前 7 个用例是原有功能测试的**逐条复刻，断言完全相同**：

| 序号 | 测试场景 | 输入 | 预期结果 | 结果 |
| --- | --- | --- | --- | --- |
| 1 | 静态文件服务 | `GET /static/hello.txt` | 200，`text/plain`，响应体与文件实际字节一致 | ✅ |
| 2 | 求和（查询串，多值+负数） | `?numbers=1,2&numbers=3&numbers=-4` | `{"numbers":[1,2,3,-4],"result":2}` | ✅ |
| 3 | 求和（非整数） | `?numbers=1,two,3` | 400 `{"error":"not an integer: two"}` | ✅ |
| 4 | 求和（表单 POST） | `numbers=10,20,30` | `{"numbers":[10,20,30],"result":60}` | ✅ |
| 5 | Cookie 解析与回显 | `Cookie: flavor=vanilla%20bean; answer=42; ignored` | 解码 + 忽略无 `=` 片段 | ✅ |
| 6 | 会话保持 | 连续两次 `/api/v1/session` | sid 相同，visits 递增 1 | ✅ |
| 7 | 登录与鉴权全流程 | `/protected` → `/login` → `/protected` | 303 → 303 → 200 | ✅ |

**另外**：我们把原始测试文件（`md5 = d2ef39b5...`，逐字节未改）直接拉起重构版跑了一次，**7/7 通过**，并用只有重构版才有的行为（无参数 sleep 返回 200、首页不含 password）确认了被测的确实是重构版。

> 但我们**没有**把这个文件纳入常规测试集——它硬编码 8080，一旦该端口被其它进程占用，结果就不可信。这正是我们重写测试的原因，把它留在套件里等于自打耳光。

## 六、第三轮：把单元测试行覆盖率补到 100%

### 6.1 补之前的缺口长什么样

第三轮开始前，`test_unit.py` 单独跑的行覆盖率是 **74%**：

| 模块 | 补之前 | 补之后 |
| --- | --- | --- |
| `config.py` | 100% | 100% |
| `sessions.py` | 99% | 100% |
| `http_core.py` | 97% | 100% |
| `handlers.py` | 70% | 100% |
| `http_server.py` | **50%** | 100% |
| **合计** | **74%（553 行中 142 行未覆盖）** | **100%** |

关键不在这个数字，而在**缺口的分布**：`http_server.py` 独占了 91 行未覆盖，`handlers.py` 又占 46 行。把它们逐行读一遍，缺口几乎全部落在同一类代码上——**只有起一个真服务器才能走到的路径**：读超时、客户端中途断开、bind 失败、连接被取消、accept 后 setblocking 失败、`main()` 入口。

这就是原黑盒测试留下的结构性阴影的最后一块：第一轮把**业务逻辑**从网络里解耦出来了，但**连接生命周期本身**仍然只能靠功能测试从外面打。而这些恰恰是：

- 功能测试最难稳定构造的（要制造"客户端发完就走"、"accept 后 fd 失效"这类时序，黑盒几乎无从下手）；
- 线上最容易出事的（fd 泄漏、半开连接堆积、关停时残留 task）；
- 出事时最难归因的（不报错、不返回、只是慢慢耗尽 fd）。

所以这一轮补的不是"覆盖率"，是**故障路径的可测性**。

### 6.2 新增的 48 项用例

| 分组 | 数量 | 关闭的缺口 |
| --- | --- | --- |
| 报文解析剩余拒绝分支 | 2 | 请求行段数不对、头部行无冒号 |
| 会话删除 | 1 | `delete()` 从未被单测触达；补充"未知 sid 不得抛" |
| 业务纯函数剩余分支 | 3 | 空片段跳过、合法 sec（含边界 0 与上限值）、`/static/` → `index.html` |
| Router 各端点 | 14 | 首页/求和/sleep/session/登录/登出/protected 的成功与拒绝分支，**此前全部只有功能测试覆盖** |
| 静态文件（router 层） | 6 | content-type 猜测与回退、目录 index、穿越 403、缺失 404、**读文件 I/O 失败 500** |
| `read_request` socket 边界 | 4 | 头部截断、头完整但超限、**body 分包到达**、body 截断 |
| `ConnectionHandler` 故障路径 | 7 | **读超时 408**、解析错误按自带状态码回、router 抛 HttpError、读阶段意外异常兜 500、**客户端断开**、**取消时关 socket**、有效 sid 复用并续期 |
| `HttpServer` 启停 | 6 | **bind 失败给人话**、port=0 分配、accept 循环、关停取消在途连接、setblocking 失败丢弃该连接、`serve()` |
| 命令行入口 | 5 | `parse_args` 三种、`main()` 的 Ctrl-C 路径、**`python http_server.py` 作为脚本可执行** |

几条值得单独说的：

- **`test_start_reports_readable_error_when_port_is_taken`** —— 它构造的正是我们在任务一里踩过的场景：端口被别的进程占着。断言 `RuntimeError` 里带 `cannot bind` 和端口号，而不是原代码那样甩一堆 asyncio 栈帧让人误判成代码 bug。整个测试用 `port=0` 先占一个内核分配的端口再去撞，**不碰 8080**，因此在任何机器上都可跑。
- **`test_cancelled_connection_still_closes_its_socket`** —— `CancelledError` 不是 `Exception`。如果 `client.close()` 写在 `except` 里而不是 `finally` 里，关停时每条在途连接都会漏一个 fd，且**测不出、不报错**。这条用例直接断言 `left.fileno() == -1`。
- **`test_read_timeout_returns_408`** —— 通过注入 `read_timeout_seconds=0.05` 在 0.05 秒内测完 slowloris 防护。这正是[第七节](#七我们承认的测试盲区)原先登记的盲区："要测 15 秒读超时就得等 15 秒"——注入配置后它不再成立。
- **`test_module_runs_as_a_script`** —— `python http_server.py` 是文档写明的用法，用 `runpy.run_path(..., run_name="__main__")` 真的执行一遍 `__main__` 入口（`asyncio.run` 被替换为立即抛 `KeyboardInterrupt`，因此不会真起服务）。这样 100% 是跑出来的，而不是靠 `# pragma: no cover` 把这行从分母里划掉。

### 6.3 100% 覆盖率不等于测试有效——所以我们做了变异检验

行覆盖率只证明"这行被执行过"，**不证明"改坏了会被发现"**。一个全是 `assert True` 的测试集照样能拿 100%。

为了证伪"这 48 条只是把行走了一遍"，我们把重构版复制一份，逐个注入故意的 bug，看单测是否报警：

| 注入的 bug | 结果 |
| --- | --- |
| `sessions.delete()` 改成空操作 | **2 项失败** ✅ |
| `_handle_protected` 的权限判断恒假（任何人都能进） | **1 项失败** ✅ |
| 静态文件读取不再捕获 `OSError` | **1 项失败** ✅ |
| `handle()` 的 `client.close()` 从 `finally` 移走（fd 泄漏） | **8 项失败** ✅ |
| `start()` 的 bind 失败不再转成可读的 `RuntimeError` | **1 项失败** ✅ |
| sleep 端点不再拒绝非 GET/POST | **1 项失败** ✅ |
| 无注入（基线） | 103 项全过 ✅ |

**6/6 全部被捕获**。其中"移走 `client.close()`"一项让 8 条用例失败并让耗时从 0.27s 涨到 21.56s——fd 泄漏在测试里表现为大面积超时，这正是我们希望它在 CI 里发出的声音。

### 6.4 复现方式

在仓库根目录执行：

```bash
# macOS 的 Homebrew Python 受 PEP 668 管控装不了全局包，用临时 venv 即可：
python3 -m venv --system-site-packages covenv && ./covenv/bin/pip install coverage

./covenv/bin/python -m coverage run --source=. --omit='tests/*,http_server_ori.py' -m pytest tests/test_unit.py -q
./covenv/bin/python -m coverage report -m
```

`--omit` 里的 `http_server_ori.py` 是留作对照的原始问题代码，不是重构产物；漏掉这个参数，它的 255 行会被算进分母，报出来的就是 68% 而不是 100%。

日常跑测试不需要 coverage，`./run-tests.sh` 即可。

### 6.5 这一轮的代价与保留意见

诚实登记，避免把 100% 当成免检标志：

- **100% 是行覆盖，不是分支覆盖**，更不是路径覆盖。`coverage --branch` 我们没跑。
- **两条用例用了 mock**：`test_serve_forever_drops_client_it_cannot_configure` 替换了事件循环的 `sock_accept`，`test_unexpected_error_while_reading_request_returns_500` 替换了模块级的 `read_request`。这两条防御分支（accept 后 fd 失效、读阶段意外异常）无法用真 socket 稳定构造，只能如此；它们对实现细节的耦合比其它用例高，重构这两处时可能需要同步改测试。
- **单元测试耗时从 0.06s 涨到 0.27s**。多出来的时间几乎全在三条用例上：读超时要真等 0.05 秒，body 分包要真等 0.05 秒让对端先消费头部，取消用例要真等 0.05 秒进入读等待。仍然不起进程、不绑固定端口，可并行。
- **`test_static_unreadable_file_returns_500_not_400` 在 root 用户或 Windows 下会自动跳过**（chmod 000 对 root 无效），跳过时该分支实际未被覆盖——也就是说，**以 root 身份跑出来的覆盖率不是 100%**。

## 七、我们承认的测试盲区

诚实登记，避免让人误以为覆盖是完整的：

- **并发正确性无测试**。我们实测过 `visits` 的读-改-写今天是原子的（因为中间恰好没有 `await`），但没有为它写回归测试——因为它的正确性来自"没有 await"这个结构性事实，而不是任何断言能锁住的东西。**加一个 await 就会 100 → 1**，且不报错、无日志。这条我们只能靠代码评审守住，100% 行覆盖对它无能为力。
