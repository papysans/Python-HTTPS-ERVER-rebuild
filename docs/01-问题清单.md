## 二、重点问题详述

我们挑五条最能说明"代码能跑 ≠ 代码质量好"的展开。

### 问题 1（致命）——测试全绿，但测的不是这份代码

这是我们找到的最严重的问题，而且它不在源码的任何一行里。

原测试用 `subprocess.Popen` 拉起真进程，硬编码 `BASE_URL = "http://127.0.0.1:8080"`。我们在本机连续跑三次，得到**三种结果**：

| 运行 | 结果 | 实际被测对象 |
| --- | --- | --- |
| 第 1 次 | 4 失败 / 3 通过 | 前 4 个打到外部程序，后 3 个打到子进程 |
| 第 2 次 | **7 全部通过（0.06s）** | **一个残留的旧孤儿进程**，本次子进程已 EADDRINUSE 崩溃 |
| 第 3 次 | 7 全部失败（0.05s） | 全部打到本机另一个占用 8080 的程序 |

根因是 `wait_for_server()` 每轮**先** `poll()` **后** `connect`。子进程启动需几十毫秒，第一轮 `poll()` 必然返回 `None`（"看起来还活着"），而 `create_connection` 立刻连上了**占位者**并 return。子进程带着 `EADDRINUSE` 崩溃栈死掉，但栈被灌进 `subprocess.PIPE` 后永远不会打印——只有 `poll() is not None` 那条分支才会 `communicate()` 抛错，而它被竞态绕过了。

> **最危险的不是变红，是变绿。** 第 2 次运行 7/7 通过、0.06 秒，被测对象是一个早该死掉的旧进程。**你可以把 `route()` 整个删掉，测试照样全绿。**

因果链是唯一的：核心逻辑全部耦合全局状态 → 只能从 8080 这个唯一入口触达 → 被迫写进程级 e2e → 依赖不可控的全局资源 → **测试信号丧失证伪能力**。

**可测试性缺失不是"不方便"，它直接摧毁了测试的证据能力。**

### 问题 2（致命）——会话固定，实测可无密码拿到 admin

`handle_login` 在**现有 session 对象**上写 `user`（L246），用**原 sid** 发 Set-Cookie（L248），全程没有换过 sid。实测攻击链：

```
攻击者先领一个合法 sid:  t7EZfEmPsy2Cj14ptgx0...
受害者带着该 sid 登录  -> 303 | Set-Cookie: sid=t7EZfEmPsy2Cj14ptgx0...  (sid 未变)
攻击者重放该 sid       -> /protected: 200 "You are logged in."
SESSIONS[sid] 现在是:   {'user': 'admin'}
```

攻击者访问任意 URL 领一个 sid，通过子域 cookie 注入或明文 HTTP 中间人种给受害者，受害者一登录，该 sid 就带上了 `user=admin`。**无需密码，权限完全绕过。**

### 问题 4 + 3（高 + 致命）——一行死代码暴露的三重失败

```python
sec = source.get("sec")   # 缺参时返回 None
sleep_sec = 1             # 这行永远不会执行
if len(sec) > 0:          # len(None) -> TypeError
    sleep_sec = int(sec[0])
```

实测 `GET /api/v1/sleep`（不带参数）→ `400 bad request: object of type 'NoneType' has no len()`。

这一处同时暴露三层失败，值得逐层看：

1. **它是死代码**：`sleep_sec = 1` 想表达"默认睡 1 秒"的业务意图，但它想兜底的那个场景**恰恰是崩溃的场景**。这行代码是一句谎言。
2. **表达不一致是根因**：相邻 20 行的 sum 分支用的是正确惯用法 `source.get("numbers") or []`（L334），sleep 却退化成 `.get()` + `len()`。同一文件、同一个 `source` 对象、两种取值范式，其中一种是坏的。
3. **错误处理把它藏了起来**（问题 3）：这是一个 `TypeError`，是**服务端 500 级缺陷**，却被 L388 的 `except Exception` 报成 `400 Bad Request`。**任何监控 5xx 率的告警永远不会响——服务在坏，指标却是绿的。** 同时 Python 异常原文被回显给客户端。

我们实测了信息泄露的最坏一例（`static/secret.txt` 权限 000 时）：

```
400 | bad request: [Errno 13] Permission denied:
     '/<服务器部署根路径>/static/secret.txt'
```

**绝对文件系统路径直接回给了攻击者。**（此处已按需脱敏，实测输出为完整真实路径。）

### 问题 8（高）——过期会话被复活，过期策略形同虚设

`get_session` 命中后**立刻**刷新 `last_seen`（L116），而 `cleanup_sessions()` 在 `route()` 里（L303）**更晚**才执行。于是一个早已过期的会话，在被判定过期之前就已经被续期了。

**只要客户端一直带着旧 sid 访问，会话就永不过期。** 判定过期必须先于刷新——这是顺序问题，不是逻辑问题，所以读代码时极容易滑过去。

### 问题 12（高）——早退分支绕过尾部兜底

`route()` 里 6 个分支走 `response = ...` 赋值后**流经** L378-379 的 Cookie 兜底；但 L348（sum 校验失败）和 L356（sleep 方法不允许）走 `return`，**直接跳过**。实测：

```
早退分支: DELETE /api/v1/sleep -> 405，无 Set-Cookie
正常分支: GET /nope            -> 404，Set-Cookie: sid=tpc7Alk...
```

"函数尾部统一后处理"是一种**隐式契约**，而 `return` 是打破它的合法语法——编译器和测试都不会提示你漏了什么。这不是笔误，是这个结构必然孕育的缺陷：只要有人再加一个早退分支，就会再复现一次。

## 三、经核实**不成立**的问题（AI 报了，但我们驳回）

这一节记录我们**没有采纳**的 AI 结论。它们看起来都很像问题。

| AI 的说法 | 核实结论 | 我们的证据 |
| --- | --- | --- |
| `safe_static_path` 存在路径穿越漏洞 | **不成立，防护有效** | `/static/../http_server.py`、`/static/%2e%2e/...`、`/static//etc/passwd` 全部返回 `None` → 403。`Path / "/abs"` 会被 `resolve()` + `relative_to()` 拦下 |
| 符号链接可绕过静态目录限制 | **不成立** | 实建 `static/escape_link -> /etc/passwd`，仍返回 `None`。因为 `resolve()` 会跟随符号链接，逃逸后 `relative_to` 必然失败 |
| 解析失败会泄漏一个 session | **不成立** | `get_session` 在 L171，位于所有 `raise`（L139/L148）**之后**。实测解析失败后 `SESSIONS` size = 0 |
| sid 可做 CRLF 响应头注入 | **当前不可利用** | `get_session` 仅当 sid 已是 `SESSIONS` 的 key 才回显，而 key 只由 `token_urlsafe` 生成。注入 `sid=abc%0d%0aX-Injected:%20yes` 后响应头无 `X-Injected`。**但** `to_bytes` 从不校验头部值，属潜伏风险 |
| 超大整数 `int('9'*100000)` 可致 DoS | **不成立** | CPython 4300 位限制使其直接抛 ValueError，走正常 400 分支 |
| `visits` 的读-改-写存在并发竞态 | **今天不成立，但纯属侥幸** | 实测 100 并发 → `visits = 100`（正确）。因为 `parse_request`/`get_session` 全是同步函数，中间没有 `await`。**但**给它加一个 `await`，实测立刻变成 `visits = 1`（99% 更新丢失）。代码里没有任何注释或结构在保护这个不变量 |

最后一条我们认为最有价值：**它今天是对的，但对得没有道理。** 任何一次自然演进（session 存 Redis、加异步鉴权）都会让计数凭空蒸发，而且不报错、无日志、测试也测不出来。我们把它记为"潜伏缺陷"而非"当前缺陷"。

## 四、我们的判断：多数问题共享一个主要结构性根因

这 30 条并非彼此完全独立，其中多数可维护性与可测试性问题共享一个主要结构性根因。真正该问的问题是：

> **为什么这个文件里几乎每个函数都在"顺手多做一点"？**

答案不是"作者不够仔细"，而是：**这个文件没有任何显式的层边界。**

协议解析、会话状态、路由分发、业务处理、表示层全部平铺在同一个命名空间里，函数之间**没有不可穿越的接缝**。于是：

- **职责必然漂移** —— `parse_request` 能直接摸到 `SESSIONS`，所以它就摸了（问题 6）；`route` 能直接调 `cleanup_sessions`，所以它就调了（问题 21）。**当没有墙时，捷径总会被走。**
- **命名必然退化** —— 命名要从"层"里取得语义（`XxxParser`、`XxxStore`）。没有层，就只剩 `do_` 和 `type1/type2` 这种无所指的占位符（问题 19、20）。
- **一致性必然瓦解** —— 没有层，就没有"这一层的错误怎么表达"的统一位置，于是错误格式长出三套、头解析长出两份且已分叉（问题 24）、取参范式长出两种且其中一种是坏的（问题 4）。
- **测试必然失效** —— 没有层，业务逻辑就只能从网络入口触达，于是测试被迫赌一个全局端口（问题 1）。

所以重构的排序不应是"从表格第一行往下改"，而应是**先划层**。层一旦立起来，一大半问题会自动消解。这直接决定了我们任务二的方案。

> **先改名字而不划层，是在给注定要被拆掉的墙刷漆。**
