# 人工初审问题与初步重构设想

> 这是小组第一次阅读原始代码后整理的初步记录。当时主要关注比较明显的问题，部分判断还需要后续测试验证。

## 一、初步发现的问题

1. **代码都集中在一个文件中**

   网络连接、请求解析、路由、业务处理和 session 管理混在一起。代码虽然可以运行，但阅读和修改比较困难。

2. **路由函数太长**

   `route()` 中包含很多 `if/elif`，求和、sleep、登录等功能都写在同一个函数里。以后增加接口时还要继续添加分支，容易漏掉公共处理。

3. **全局变量较多**

   端口、静态目录和 session 都依赖全局变量。测试时不方便替换，也可能造成不同测试之间互相影响。

4. **输入检查不够完整**

   例如 sleep 接口缺少 `sec` 时可能报错，传入很大的时间也没有限制。部分异常输入没有清楚的处理规则。

5. **异常处理比较粗糙**

   程序把很多异常统一当成 400 返回，有可能把服务器自身的问题当作客户端错误，同时还可能把内部异常信息直接返回给用户。

6. **session 管理比较混乱**

   session 使用全局字典保存，创建、续期和清理逻辑混在一起。初步看还存在数量没有上限、登录后 sid 不变化等风险。

7. **测试不够稳定**

   原测试固定使用 8080 端口。如果端口已被占用，可能影响测试结果。当前测试也主要验证正常请求，对异常情况覆盖较少。

8. **缺少必要的日志和资源限制**

   代码主要使用 `print`，出现错误时不容易定位。同时请求读取没有明显超时，并发连接数量也没有限制。

## 二、准备进一步确认的问题

- 占用 8080 端口后，原来的测试是否还能正确判断服务器启动失败。
- sleep 不传参数或传入很大数值时，服务器实际怎样响应。
- 登录前后是否一直使用同一个 sid。
- 发送不完整请求后，连接是否会长时间不释放。
- 连续发送匿名请求后，session 数量是否不断增加。

## 三、初步重构思路

暂时准备把原来的单文件代码按职责拆成下面几部分：

```text
http_server.py   负责网络连接和整体请求流程
http_core.py     负责 HTTP 请求解析和响应生成
handlers.py      负责路由和具体接口业务
sessions.py      负责 session 的创建、查找和清理
config.py        集中保存端口、目录和各种限制
```

初步原则是：只有 `http_server.py` 直接处理 socket，其他模块只处理普通的请求和响应对象。

## 四、初步实施顺序

1. 先补几个测试，确认端口、sleep、session 和异常处理方面的问题。
2. 把请求和响应的数据结构从原文件中抽出来。
3. 把 session 全局字典封装成单独的管理类。
4. 将长路由函数拆成路由表和多个小的 handler。
5. 再补充超时、并发限制、日志和统一错误处理。
6. 每完成一步都运行原有功能测试，避免正常功能被意外改坏。

## 五、暂时不处理的内容

- 不引入 Web 框架，仍然使用 `asyncio` 和 socket 完成练习。
- 暂时不实现 keep-alive 和 chunked 请求体解析。
- 暂时不处理密码系统、登录限速和 CSRF 等生产环境功能。
- 静态文件流式传输先作为后续优化，不放在第一轮重构中。

## 六、初步结论

目前认为主要问题不是某一行代码写错，而是不同职责混在一起，导致代码难测试、难修改。第一步应先拆分结构并建立稳定测试，再逐项修复已经确认的问题。
