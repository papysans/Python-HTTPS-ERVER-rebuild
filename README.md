# asyncio HTTP Server — 重构版

一个用 `asyncio` + `socket` 手写的 HTTP 服务器。**没有任何第三方依赖**，标准库跑通。

它的来历：原本是一份刻意写成「能跑，但难维护、难测试」的单文件代码（426 行），作为编码风格练习的反面教材。本仓库是对它做完整代码评审后的重构产物——**代码本身只是结果，`docs/` 里的评审与重构过程才是重点**。

两端结果一致：`static/hello.txt` 已通过 `.gitattributes` 钉死为 LF，且字节级断言按文件实际字节计算，不受 Windows CRLF 影响。符号链接用例在无权限的 Windows 上会自动 `skip`（开启开发者模式后可跑）。

## 功能

- 静态文件服务（含路径穿越防护，符号链接逃逸同样拦截）
- API 接口：`/api/v1/sum` 整数列表求和、`/api/v1/sleep`、`/api/v1/session`
- querystring / cookie / `application/x-www-form-urlencoded` 表单解析
- 会话管理（容量上限 + LRU 淘汰 + 过期回收 + 登录时轮换 sid）
- 简单用户名密码认证（凭据硬编码，这是原始命题的要求）

## 快速开始

```bash
python3 http_server.py                 # 监听 127.0.0.1:8080
python3 http_server.py --port 0        # 由内核分配端口
python3 http_server.py --log-level DEBUG
```

Windows 上把 `python3` 换成 `python` 或 `py` 即可，例如 `py http_server.py --port 0`。

### 三个最值得看的发现

**1. 原有的 7 个功能测试是假的安全网。** 它们硬编码打 `127.0.0.1:8080`，而 `wait_for_server()` 先 `poll()` 后 `connect`。同一份代码连跑三次得到三种结果，其中一次 **7/7 全绿、0.06 秒，被测的却是一个残留的旧进程**——把 `route()` 整个删掉，测试照样绿。

> 不只是"能跑 ≠ 质量好"，而是**"测试全绿 ≠ 代码是对的"**。当可测试性缺失时，测试本身也会沦为"看起来能跑"的一部分。

**2. 一行死代码同时暴露三重失败。**

```python
sec = source.get("sec")   # 缺参时返回 None
sleep_sec = 1             # 这行永远不会执行
if len(sec) > 0:          # len(None) → TypeError
```

`sleep_sec = 1` 想表达"默认睡 1 秒"，但**它想兜底的场景，恰恰就是崩溃的场景**。而 `except Exception` → 400 把这个 500 级缺陷伪装成了客户端错误——**监控 5xx 率的告警永远不会响**。

**3. 跨模型审查抓到了同源模型的集体盲区。** `get_session` 先刷新 `last_seen`、清理逻辑后判定过期 → 过期会话被复活，过期策略形同虚设。三个同源审查者都盯着紧挨着的"会话表无上限"，**集体漏掉**了这个顺序 bug；换一个模型家族的审查者一眼看到。

### 两点声明

- **[行为变更清单](docs/02-重构方案.md#八刻意的行为变更清单)**：重构要求"不改变原始业务行为"，但 15 处"行为"本身就是缺陷（sleep 崩溃、会话固定、HEAD 返回 body……）。修缺陷必然改行为，因此逐条登记原行为/新行为/理由，而非混在重构里静默改掉。
- **[我们没有做的事](docs/02-重构方案.md#九我们没有做的事)**：CSRF 防护、登录限速、密码哈希、静态文件流式传输、会话惰性创建**均未实现**，已显式登记，避免误以为已解决。

## 已知限制

这是一个练习项目，**不适合生产使用**。除上述未做项外，还包括：仅 `Connection: close`（无 keep-alive）、不支持 chunked（明确回 501）、静态文件一次性读入内存。

## 环境

Python 3.10+。测试需要 `pytest`，其余仅用标准库。

跨平台：共 80 项测试，在 macOS 上验证全部通过；已针对 Windows 做兼容性适配（LF 检出、字节级断言按实际文件字节计算、无符号链接权限时相关用例自动 `skip`，需开启开发者模式或以管理员身份运行才会执行）。所有文本文件经 `.gitattributes` 统一按 LF 检出，因此行尾差异不会影响测试结果。
